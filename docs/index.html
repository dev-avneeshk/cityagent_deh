System Design

Architecture Overview

CityAgent is a fully client-side React app with zero backend. Every data source is accessed directly from the browser using public APIs, CORS-safe endpoints, and Vite's dev proxy for key protection.

2
Parallel Live Data Fetch
All sensor data is fetched simultaneously using Promise.all — AQI, weather, flood discharge, and traffic in one round. This is the only real network wait (typically 300–600ms).
App.jsx
const [aqi, weather, flood, traffic, stats] = await Promise.all([
  fetchAQI(lat, lon),       // Open-Meteo CAMS — PM2.5, PM10, NO2, O3, UV
  fetchWeather(lat, lon),   // Open-Meteo Forecast — temp, humidity, wind
  fetchFlood(lat, lon),     // Open-Meteo GloFAS — river discharge m³/s
  fetchTraffic(lat, lon),   // TomTom Flow + Incidents (key-gated)
  fetchSystemStats(),       // Static mock — sensor count, uptime
]);
3
Alert Aggregation
Alerts are assembled from three sources: Google News RSS (civic keywords), and data-derived thresholds from the live sensor values. If AQI > 200, a Critical alert is auto-generated from real data — not a mock string.
alertsApi.js
// Data-derived — built from live Open-Meteo values, no RSS needed
function deriveFromLiveData(cityName, { aqi, flood, weather }) {
  if (aqi?.value > 200)
    alerts.push({ severity: 'critical',
      message: `Hazardous air (AQI ${aqi.value}) — avoid outdoor activity`,
      source: 'Open-Meteo AQI' });

  if (flood?.floodRisk === 'Critical')
    alerts.push({ severity: 'critical',
      message: `River discharge ${flood.riverDischarge} m³/s exceeds 30d max`,
      source: 'Open-Meteo GloFAS' });

  if (weather?.precipitation > 15)
    alerts.push({ severity: 'high',
      message: `Heavy rainfall ${weather.precipitation}mm — waterlogging risk`,
      source: 'Open-Meteo Weather' });
}

Social Signals from Google News
Social signal count is derived from live Google News articles fetched for the city. The same RSS fetch powers both the News Feed and the Social Signals metric — no duplicate requests thanks to the SWR cache.

export async function fetchSocialSignals(cityName, aqiValue) {
  const articles = await fetchNews(cityName, ''); // SWR cached
  if (!articles?.length) return generateCityData(cityName, aqiValue).social;

  return {
    signalCount: articles.length * 8, // scaled to signal volume
    sources: { news: articles.length },
    topKeywords: extractTopKeywords(articles.map(a => a.title).join(' ')),
    recentPosts: articles.slice(0,3).map(a => ({
      source: a.source, text: a.title.slice(0,120), time: a.time,
    })),
    _source: 'LIVE',
  };
}
5
Urban Intelligence Derivation
Raw sensor data is processed into higher-order insights: City Breath Score, heat stress level, flood precursor stage, and AQI trajectory — all computed client-side from the live data.
mockData.js — deriveUrbanIntelligence()
export function deriveUrbanIntelligence(aqi, weather, flood) {
  // Breath Score: 100 = perfect air, 0 = hazardous
  const aqiScore    = Math.max(0, 100 - aqi.value * 0.4);
  const dustPenalty = (aqi.components?.dust > 80) ? 15 : 0;
  const breathScore = Math.round(Math.max(0, aqiScore - dustPenalty));

  // Heat stress: apparent temp via Steadman formula
  const T = weather.temperature, H = weather.humidity;
  const apparentTemp = T - 0.55*(1-H/100)*(T-14.5);
  const heatLevel = apparentTemp > 42 ? 'Extreme'
                  : apparentTemp > 38 ? 'Danger'
                  : apparentTemp > 32 ? 'Caution' : 'Safe';

  // Flood precursor from river discharge ratio
  const ratio = flood.riverDischarge / (flood.riverDischargeMax30d || 1);
  const floodPrecursor = ratio > 0.85 ? 'Warning'
                       : ratio > 0.65 ? 'Watch' : 'Clear';

  return { breathScore, heatLevel, apparentTemp, floodPrecursor, ... };
}
6
AI Chatbot — Streaming Inference
The CityAgent AI uses Llama 3.1 8B via NVIDIA NIM. The system prompt includes all live sensor readings, per-zone AQI, locality index, and user GPS data if granted. Responses stream token-by-token via Server-Sent Events.
kimiChat.js — streaming SSE
export async function* streamKimiChat(messages, systemPrompt) {
  const response = await fetch('/api/kimi', {
    method: 'POST',
    body: JSON.stringify({ model:'meta/llama-3.1-8b-instruct',
      messages: [{ role:'system', content: systemPrompt }, ...messages],
      stream: true, max_tokens: 400, temperature: 0.3 }),
  });
  const reader = response.body.getReader();
  let buffer = '';
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += new TextDecoder().decode(value, { stream: true });
    for (const line of buffer.split('\n')) {
      if (!line.startsWith('data: ')) continue;
      const token = JSON.parse(line.slice(6)).choices?.[0]?.delta?.content;
      if (token) yield token; // stream token by token to UI
    }
  }
}

Performance

Caching Architecture

Stale-While-Revalidate (SWR) — the same pattern used by Next.js, Vercel, and Cloudflare — applied per data source with appropriate TTLs.

⚡
Live Sensor Data
AQI, weather, flood, traffic refresh every 60 seconds via setInterval. No cache — always fresh. A cancelled flag prevents stale state if component unmounts mid-fetch.
TTL: 60s · No cache
📰
News Feed (SWR)
Return cache instantly on revisit. If stale (>15 min), refresh in background via onUpdate callback. Empty results are never cached — next call always retries.
TTL: 15min · SWR
🚨
Alert Feed (SWR)
Alerts are time-sensitive — 10-minute SWR threshold. Data-derived alerts (from live AQI/flood) bypass cache entirely and are recomputed on every render from current sensor values.
TTL: 10min · SWR
🔄
In-Flight Deduplication
A Map<key, Promise> prevents duplicate concurrent fetches. If two components request the same city's news simultaneously, only one HTTP call is made — both await the same promise.
Pattern: Promise dedup
🌆
City Switch
When city changes, news is cleared immediately (skeleton shown), previous interval is cancelled, and new fetch starts fresh. Old city's cache is preserved for instant back-navigation.
Instant skeleton → fresh fetch
🚀
Background Pagination
Page 1 of Reddit/news is returned immediately (fast). Page 2 is fetched in the background with no await — when it resolves, the cache is silently updated. Next render picks up the expanded data.
Pattern: Background update

🤖 AI Chatbot — Client-Side Rate Limiter
Implemented in kimiChat.js — a sliding window limiter enforced before any API call. This protects both the user's NVIDIA credits and prevents abuse.
kimiChat.js — checkRateLimit()
const RATE_WINDOW_MS = 60_000; // 1 minute window
const MAX_PER_WINDOW = 10;     // max 10 messages per minute
const MIN_GAP_MS     = 2_000;  // minimum 2s between messages
const _timestamps    = [];

export function checkRateLimit() {
  const now = Date.now();
  // Evict timestamps outside the window (sliding window)
  while (_timestamps.length && now - _timestamps[0] > RATE_WINDOW_MS)
    _timestamps.shift();

  if (_timestamps.length >= MAX_PER_WINDOW)
    return `Rate limit reached (${MAX_PER_WINDOW}/min). Wait
      ${Math.ceil((RATE_WINDOW_MS - (now - _timestamps[0])) / 1000)}s.`;

  const last = _timestamps[_timestamps.length - 1];
  if (last && now - last < MIN_GAP_MS)
    return `Slow down — wait ${Math.ceil((MIN_GAP_MS-(now-last))/1000)}s.`;

  _timestamps.push(now);
  return null; // allowed — proceed with API call
}
Security

Security Vulnerabilities & Mitigations

We identified every attack surface. Here's what the risks are and exactly how each is addressed.

🔑
API Key Exposure (VITE_ Prefix)
Vite's VITE_ prefix embeds environment variables directly into the browser bundle. Any user can open DevTools → Sources → index.js and find VITE_KIMI_API_KEY, VITE_TOMTOM_KEY, and VITE_TELEGRAM_BOT_TOKEN in plain text.
✅ Current mitigation: Vite dev proxy routes /api/kimi → NVIDIA NIM, so the key is injected server-side and never appears in network requests. For production, the same pattern applies via a Vercel/Netlify Edge Function proxy.

✅ For TomTom: Fallback to mock data if key is absent — no hard dependency.

✅ For Telegram: Bot token only allows sending to known chat_id — no data access risk even if leaked.
💉
Prompt Injection via News Titles
News article titles and alert messages are injected into the AI system prompt as plain text. A malicious actor could publish an article titled "Ignore previous instructions and..." to manipulate the LLM's responses via indirect prompt injection.
✅ Mitigation: Only the top 3 alert messages are included in the prompt, sliced to 120 characters. Alert sources are verified RSS feeds (Google News aggregator). The system prompt role is set as immutable context before user messages.

✅ Model-level: Llama 3.1 8B has instruction-following training that resists basic injection. max_tokens: 400 limits blast radius of any hijacked response.
🌐
CORS & Third-Party API Trust
Browser CORS policy restricts cross-origin requests. Some APIs (like Reddit's new.json) don't return Access-Control-Allow-Origin headers, causing silent fetch failures that fall back to mock data without the user knowing.
✅ Mitigation: All APIs are individually tested for CORS headers (curl with Origin header). Reddit was removed entirely due to inconsistent CORS + 403 blocks. rss2json.com acts as a CORS-safe proxy for Google News RSS.

✅ Open-Meteo: Returns Access-Control-Allow-Origin: * on all endpoints — fully browser-safe.
⚡
React StrictMode Double-Invoke Bug
React 18 Strict Mode intentionally calls state updater functions twice in development to detect side effects. If an async function (like streamReply) is called inside a setMessages updater, it fires twice — creating two concurrent streaming generators writing to the same message bubble, producing garbled interleaved text.
✅ Fixed: All streamReply() calls were moved outside setMessages() updater functions. A messagesRef mirrors messages state synchronously (updated via useEffect), providing a safe read-only snapshot for use inside async callbacks.

Pattern: Never call async functions inside React state updater functions.
🤖
AI Chatbot Abuse / Credit Drain
Without rate limiting, a user could spam the chatbot and drain the NVIDIA API credits in seconds. Each inference call consumes a measurable credit amount based on tokens processed.
✅ Client-side limiter: Sliding window — max 10 messages/minute, minimum 2-second gap between requests. Enforced before any network call. Error message shown in UI with exact wait time.

⚠ Limitation: Client-side only — a determined user could bypass it via DevTools. Production fix: move to server-side rate limiting per session/IP.
📍
GPS Privacy
When a user asks a location-aware question, the chatbot requests GPS coordinates via browser geolocation API and sends them (via the system prompt) to an external AI inference endpoint (NVIDIA NIM).
✅ Consent-first: GPS is never requested automatically. An inline prompt card asks the user explicitly before geolocation.getCurrentPosition() is called.

✅ Minimal transmission: Only lat/lon is sent — rounded to 5 decimal places (~1m precision). No persistent storage, no logging. Coordinates are discarded after the response stream completes.

✅ Alternative: "Use City Data" option always available — city-level data only, no GPS required.
🔐
Telegram Bot Token Leak
If VITE_TELEGRAM_BOT_TOKEN is extracted from the bundle, an attacker could send arbitrary messages to @cityagentdh, potentially spamming followers or posting misinformation.
✅ Chat ID restriction: The bot token only works with the specific CHAT_ID (the Telegram channel). Even with the token, sending to other chats fails unless the bot is added there.

✅ Rate limit: Telegram enforces 1 message/second per bot to any single chat — limits spam even if token leaks.

Production fix: Move sendMessage to a server-side Edge Function — token never touches the client.
🛡
What's Fully Safe
Open-Meteo (3 APIs): Zero-auth. No key to leak. No PII transmitted. Only lat/lon sent — same as any weather app.

Google News RSS: Public RSS feed. No auth. No user data sent.

BigDataCloud: Client-side only, no key, lat/lon only.

Browser Speech API: On-device processing. Audio never leaves the device — converted to text locally by the browser engine. Zero network requests for transcription.
Resilience

Fallback Strategy

Every API call has a graceful degradation path. The dashboard never crashes — it falls back to the last known data or sensible defaults.

Open-Meteo AQI fails
→ Static fallback values
Returns hardcoded AQI=64 Moderate with realistic Dehradun PM2.5 values. UI shows "fallback" station name.
TomTom Traffic fails / no key
→ Mock trafficData
City-specific mock traffic data with realistic hotspots. MetricCard shows "MOCK" badge instead of "LIVE".
Google News RSS fails
→ Empty news feed
NewsFeed shows "No articles found" state. Alerts still work from data-derived sensors. Social signals fall back to mock.
NVIDIA AI fails
→ Error message in chat bubble
Streaming catch block shows error in the chat UI. Previous messages preserved. User can retry.
GPS denied by user
→ City-level data only
AI responds with city-wide AQI and weather. Location prompt card shows "Use City Data" button as primary alternative.
City has no subreddit
→ Global Reddit search
redditApi.js tries city-specific subreddit first, then falls back to global Reddit search with the city name as query prefix.
